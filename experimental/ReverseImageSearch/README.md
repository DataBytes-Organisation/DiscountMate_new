# Reverse Image Search — DiscountMate Experimental

**Status:** Integrated (experimental service, live in development)

A visual product-matching service for DiscountMate. Users upload or photograph a grocery product and receive a ranked list of visually similar items from the Coles catalogue, each with live pricing data.

---

## How it works

1. User uploads an image via the camera button in the search bar.
2. The Express backend (`/api/reverse-image-search`) proxies the upload to this FastAPI service.
3. The image is passed through **DINOv2 (ViT-B/14)** with 5 test-time augmentations (normal, blur, saturated, flipped, centre-zoomed) to produce 5 query vectors.
4. Each vector is queried against a prebuilt **FAISS HNSW index** (768-dim, ~5× entries per product — one per augmentation variant built at index time).
5. Results are deduplicated by `mongo_id`, re-scored with a hybrid TTA strategy (`0.7 × max + 0.3 × mean`), and returned as a ranked list.
6. The Express backend enriches results with **live pricing** from the MongoDB `product_pricings` collection (Coles, Woolworths, IGA).
7. Product images are resolved from CDN URLs stored in MongoDB; local `Scraped_images/` is used as a fallback for legacy entries.

## Prerequisites

- Python 3.9 or 3.10+
- **`index.faiss` and `metadata.json` must be generated first** by running `build_faiss_index.ipynb` (see [Rebuilding the index](#rebuilding-the-index) below). The service will refuse to start if either file is missing.
- `Scraped_images/` is produced by the same notebook run and required for legacy local-image fallback.

Install dependencies:

```bash
pip install -r requirements.txt
```

`xFormers is not available` warnings from PyTorch are safe to ignore — the model loads correctly without it.

## Running the service

```bash
cd experimental/ReverseImageSearch
python3 -m uvicorn api:app --host 127.0.0.1 --port 8001
```

The service will be available at `http://127.0.0.1:8001`. On first startup it downloads and caches the DINOv2 model weights from Torch Hub (~330 MB).

### Running the full stack

Start all three services in separate terminals:

```bash
# 1. ML service (this folder)
cd experimental/ReverseImageSearch
python3 -m uvicorn api:app --host 127.0.0.1 --port 8001

# 2. Express backend
cd Backend
node server.js

# 3. Expo frontend
cd Frontend
npm run web
```

## API reference

### `POST /reverse-image-search?top_k=5`

Upload an image file and receive ranked matches.

| Parameter | Type | Description |
|-----------|------|-------------|
| `file` | `multipart/form-data` | Image file (JPEG, PNG, etc.) |
| `top_k` | query param (int, default 5) | Number of results to return |

**Response** — array of `SearchResult` (prices enriched by the Express backend from MongoDB):

```json
[
  {
    "rank": 1,
    "product_id": "12345",
    "mongo_id": "64a1f2e3b4c5d6e7f8a9b0c1",
    "name": "White Peppermint Sugar Free Chewing Gum Bottle",
    "similarity_score": 0.706,
    "image_url": "https://shop.coles.com.au/wcsstore/Coles-CAS/images/1/2/3/12345.jpg",
    "price_now": "$5.00",
    "price_was": "$6.50",
    "price_comparable": "$7.81 per 100g",
    "woolworths_price": "$5.50",
    "iga_price": null
  }
]
```

> **Note:** `price_now`, `woolworths_price`, and `iga_price` are populated by the Express proxy from the `product_pricings` MongoDB collection. `image_url` is a full CDN URL for catalogue products; legacy locally-scraped products return a `/images/{filename}` relative path instead.

### `GET /images/{filename}`

Serves a scraped product image from `Scraped_images/` (legacy local images only).

### `GET /health`

Returns `{"status": "ok"}` when the service is ready.

## Python 3.9 compatibility notes

Two patches were applied to make the service run on Python 3.9 (which does not support `X | Y` union syntax):

- **`api.py`** — uses `Optional[str]` instead of `str | None` in Pydantic models.
- **`notebook_runtime.py`** — `_patch_cached_dinov2_for_python39()` rewrites `float | None` annotations in the Torch Hub DINOv2 cache at startup. This runs automatically; no manual action needed.

On Python 3.10+ these patches are no-ops.

## Rebuilding the index

**This step is required before the service can run.** `index.faiss` and `metadata.json` are not included in the repository and must be generated locally.

Open and run **`build_faiss_index.ipynb`** from top to bottom (Google Colab with an A100 GPU is recommended). The notebook:

1. Downloads product images from MongoDB (CDN URLs stored in the `products` collection)
2. Extracts DINOv2 (ViT-B/14) embeddings for **5 augmentation variants** per image (normal, blur, saturated, flipped, centre-zoomed)
3. Builds a FAISS HNSW index with `METRIC_INNER_PRODUCT` (cosine similarity on L2-normalised vectors)
4. Writes `index.faiss` and `metadata.json` (each vector annotated with `mongo_id`, `product_id`, `name`, `image_url`, `augment`) to this directory

**A100-specific optimisations** in the notebook: bfloat16 autocast, `torch.compile`, TF32 matrix ops, and batch sizes tuned for 40 GB HBM.

`coles_reverse_image_search_Official_2.ipynb` is the original research notebook kept for reference — use `build_faiss_index.ipynb` for production index builds.

## Integration with the main app

| Component | Location | Purpose |
|-----------|----------|---------|
| Express proxy route | `Backend/src/routers/reverse-image-search.router.js` | Forwards uploads to FastAPI, enriches results with live MongoDB prices, normalises image URLs |
| Route mount | `Backend/server.js` at `/api/reverse-image-search` | |
| Image search context | `Frontend/app/(tabs)/ImageSearchContext.tsx` | Holds results (including `woolworths_price`, `iga_price`) across navigation |
| Context provider mount | `Frontend/app/_layout.tsx` | `ImageSearchProvider` wraps the root layout so all route groups share the same context |
| Search bar upload | `Frontend/components/layout/SearchBar.tsx` | Camera button, file picker, webcam capture |
| Results rendering | `Frontend/components/search/SearchResultsMainSection.tsx` | Renders results as `ProductCard` components with all three store prices |
| API constant | `Frontend/constants/Api.ts` → `REVERSE_IMAGE_SEARCH_API_URL` | Points to `http://localhost:3000/api/reverse-image-search` |

**Image URL resolution (frontend):** CDN URLs (`https://…`) are loaded directly in the browser. Legacy relative paths (`/images/filename.jpg`) are prepended with `REVERSE_IMAGE_SEARCH_API_URL` so Express serves them from `Scraped_images/`.

**Live price enrichment (backend):** After receiving FAISS results from FastAPI, the Express router uses each result's `mongo_id` to look up `product_code` in the `products` collection, then aggregates the latest price per store from `product_pricings` (store chains: `coles_generic`, `woolworths_generic`, `iga_generic`) in a single round-trip per store. Falls back gracefully if MongoDB is unavailable.

## Known limitations

- Image matching covers the **Coles catalogue only** — the FAISS index is built from Coles product images. Woolworths/IGA prices are shown where available in `product_pricings`, but those stores' images are not indexed.
- Index must be rebuilt manually if the product catalogue changes significantly.
- First startup is slow (~30–60 s) while DINOv2 loads into memory; subsequent requests are fast.
- `Scraped_images/`, `index.faiss`, and `metadata.json` are not committed to git due to size constraints.
- Products that have no `product_pricings` entries will show `"-"` prices in the UI — this reflects genuinely missing pricing data, not a bug.
