# Reverse Image Search — DiscountMate Experimental

**Status:** Integrated (experimental service, live in development)

A visual product-matching service for DiscountMate. Users upload or photograph a grocery product and receive a ranked list of visually similar items from the Coles catalogue, each with live pricing data.

---

## How it works

1. User uploads an image via the camera button in the search bar.
2. The Express backend (`/api/reverse-image-search`) proxies the upload to this FastAPI service.
3. The image is passed through **DINOv2 (ViT-B/14)** with 5 test-time augmentations (normal, blur, saturated, flipped, centre-zoomed) to extract feature vectors.
4. Vectors are queried against a prebuilt **FAISS HNSW index** (768-dim, 185 k entries).
5. Results are deduplicated by `product_id`, re-scored using max aggregation across augments, and returned as a ranked list.
6. The Express backend resolves product images from `Scraped_images/` and serves them directly to the frontend.

## File structure

```
ReverseImageSearch/
├── api.py                                  # FastAPI service — upload endpoint + image serving
├── notebook_runtime.py                     # Query-only runtime extracted from the notebook
├── coles_reverse_image_search_Official_2.ipynb  # Original research notebook (index building)
├── app.py                                  # Standalone Streamlit demo (optional)
├── requirements.txt                        # Python dependencies
├── index.faiss                             # Prebuilt FAISS HNSW index (not in git — see below)
├── metadata.json                           # Per-vector metadata (product_id, name, filename, etc.)
├── Master_Coles_Scrape.csv                 # Pricing data keyed by product_id
└── Scraped_images/                         # Product images scraped from the Coles catalogue
```

> `index.faiss` are excluded from git due to size. **`index.faiss` must be generated by running the notebook before the service can start** — it is not included in the repository and cannot be downloaded separately.

---

## Prerequisites

- Python 3.9 or 3.10+
- **`index.faiss` must be generated first** by running the notebook (see [Rebuilding the index](#rebuilding-the-index) below). The service will refuse to start if this file is missing.
- `metadata.json`, `Master_Coles_Scrape.csv`, and `Scraped_images/` are also required and produced by the same notebook run.

Install dependencies:

```bash
pip install -r requirements.txt
```

`xFormers is not available` warnings from PyTorch are safe to ignore — the model loads correctly without it.

## Running the service

```bash
cd experimental/ReverseImageSearch
python3 -m uvicorn api:app --host 127.0.0.1 --port 8001
```

The service will be available at `http://127.0.0.1:8001`. On first startup it downloads and caches the DINOv2 model weights from Torch Hub (~330 MB).

### Running the full stack

Start all three services in separate terminals:

```bash
# 1. ML service (this folder)
cd experimental/ReverseImageSearch
python3 -m uvicorn api:app --host 127.0.0.1 --port 8001

# 2. Express backend
cd Backend
node server.js

# 3. Expo frontend
cd Frontend
npm run web
```

## API reference

### `POST /reverse-image-search?top_k=5`

Upload an image file and receive ranked matches.

| Parameter | Type | Description |
|-----------|------|-------------|
| `file` | `multipart/form-data` | Image file (JPEG, PNG, etc.) |
| `top_k` | query param (int, default 5) | Number of results to return |

**Response** — array of `SearchResult`:

```json
[
  {
    "rank": 1,
    "product_id": "12345",
    "name": "White Peppermint Sugar Free Chewing Gum Bottle",
    "similarity_score": 0.706,
    "image_url": "/images/12345_White_Peppermint_Front_1.jpg",
    "price_now": "5.00",
    "price_was": null,
    "price_comparable": "$7.81 per 100g"
  }
]
```

### `GET /images/{filename}`

Serves a scraped product image from `Scraped_images/`.

### `GET /health`

Returns `{"status": "ok"}` when the service is ready.

## Python 3.9 compatibility notes

Two patches were applied to make the service run on Python 3.9 (which does not support `X | Y` union syntax):

- **`api.py`** — uses `Optional[str]` instead of `str | None` in Pydantic models.
- **`notebook_runtime.py`** — `_patch_cached_dinov2_for_python39()` rewrites `float | None` annotations in the Torch Hub DINOv2 cache at startup. This runs automatically; no manual action needed.

On Python 3.10+ these patches are no-ops.

## Rebuilding the index

**This step is required before the service can run.** `index.faiss` is not included in the repository and must be generated locally.

Open and run `coles_reverse_image_search_Official_2.ipynb` from top to bottom. The notebook:
1. Scrapes product images from the Coles catalogue into `Scraped_images/`
2. Extracts DINOv2 embeddings for each image
3. Builds the FAISS HNSW index and writes `index.faiss` and `metadata.json` to this directory

Requires access to the Coles catalogue scraper and sufficient disk space for `Scraped_images/`. Generation is compute-intensive and may take several hours without a GPU.

## Integration with the main app

| Component | Location | Purpose |
|-----------|----------|---------|
| Express proxy route | `Backend/src/routers/reverse-image-search.router.js` | Forwards uploads, normalises image URLs, serves images |
| Route mount | `Backend/server.js` at `/api/reverse-image-search` | |
| Image search context | `Frontend/app/(tabs)/ImageSearchContext.tsx` | Holds results across navigation |
| Search bar upload | `Frontend/components/layout/SearchBar.tsx` | Camera button, file picker, webcam capture |
| Results rendering | `Frontend/components/search/SearchResultsMainSection.tsx` | Renders results as `ProductCard` components |
| API constant | `Frontend/constants/Api.ts` → `REVERSE_IMAGE_SEARCH_API_URL` | |

Images are served by Express directly from this folder's `Scraped_images/` directory — not proxied through FastAPI — to avoid double-hop latency.

## Known limitations

- Only covers the **Coles** catalogue. Woolworths and IGA prices are not available for image search results.
- Index must be rebuilt manually if the product catalogue changes significantly.
- First startup is slow (~30–60 s) while DINOv2 loads into memory; subsequent requests are fast.
- `Scraped_images/` and `index.faiss` are not committed to git due to size constraints.
